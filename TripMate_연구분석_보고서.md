## 1. 서론

### 1.1 연구 배경

대한민국 정부는 내수 관광 진흥 및 지역 경제 활성화를 위해 매년 수백억 원 규모의 여행 보조금 및 할인 프로그램을 운영한다. 2026년 기준 주요 프로그램만 해도 **숙박세일페스타**, **지역사랑 휴가지원**, **대한민국 숙박 대전**, **온누리상품권 여행 환급** 등이 동시에 운영 중이다.

그러나 이 혜택들은 다음과 같은 이유로 실질적 활용률이 낮다.

- 각 혜택마다 **신청 경로, 조건, 마감일이 상이**하며 별도 앱/웹사이트에 분산
- 여행 **계획 단계**에서 혜택을 확인하기 어렵고, 사후에야 인지하는 경우가 다수
- 복잡한 사전 신청·영수증 제출·환급 대기 프로세스로 인한 **이탈률 증가**
- 기존 여행 예약 앱(야놀자, 여기어때)은 **상업적 커머스 중심**으로 정부 혜택 연동 미흡

### 1.2 연구 목적

본 보고서는 TripMate가 해결하고자 하는 문제의 실체를 명확히 하고, 기존 경쟁 서비스와의 차별성, 공공데이터 활용 가치, 그리고 서비스 완성도 제고를 위한 개선 방향을 체계적으로 제시하는 것을 목적으로 한다.

## 2. 국내 여행 앱 현황 분석

### 2.1 숙박 예약 플랫폼

#### 야놀자 (Yanolja)

야놀자는 국내 최대 여행·숙박 예약 플랫폼으로 모텔, 호텔, 펜션, 글램핑, 카라반 등 320만 개 이상의 숙박 시설을 보유하고 있으며, 항공·렌터카·레저까지 확장한 종합 여행 플랫폼이다. ChatGPT Enterprise를 도입해 AI 기반 고객 응대에 투자하고 있다.

**핵심 한계:**
- 정부 여행 보조금(숙박세일페스타, 지역사랑 휴가지원 등)과 **직접 연동 없음**
- 쿠폰 정책 관련 공정거래 위반 의혹으로 2026년 3월 검찰 압수수색
- **개인 취향 기반 여행지 추천** 기능 부재 (숙소 검색 중심)
- 환급 이력 관리, OCR 영수증 스캔 등 **사후 정산 기능 전무**

#### 여기어때 (Yeogi Eottae)

국내 숙박 5만여 개를 기반으로 항공, 렌터카, 입장권을 통합한 플랫폼. VR 객실 이미지와 실시간 리뷰를 강점으로 내세운다.

**핵심 한계:**
- 야놀자와 마찬가지로 **정부 혜택 연동 구조 없음**
- 소비 혜택(온누리상품권) 사용 가능 숙소 **식별 기능 부재**
- 여행 코스 추천이 아닌 **숙소 단위 검색** 중심
- 비수도권 강조 또는 인구감소지역 특화 콘텐츠 **없음**

---

### 2.2 정보 탐색형 플랫폼

#### 네이버 지도 / 카카오맵

두 플랫폼 모두 지역 탐색 및 길 안내에 최적화되어 있으며, 카카오맵은 실시간 버스 위치 및 환승 알림, 네이버 지도는 영어 지원과 정기 로드뷰 업데이트가 강점이다.

**핵심 한계:**
- **예약 기능 제한적**, 숙박·교통 예약으로 이어지는 전환율이 낮음
- 혜택 정보가 **광고성 게시물과 혼재**되어 신뢰도 저하
- 개인화된 **여행 일정 설계 기능 없음**
- 정부 보조금 정보와의 연계 **전무**

---

### 2.3 정부 공식 플랫폼

#### 대한민국 구석구석 (Visit Korea App)

한국관광공사(KTO)가 운영하는 공식 국내 여행 정보 앱. 여행지 추천, 축제 정보, 디지털 관광 주민증 등을 제공한다.

**핵심 한계:**
- UI/UX가 노후화되어 **사용자 이탈률 높음**
- 정보 열람 중심 앱으로 **예약·결제 연동 없음**
- 혜택 정보가 별도 페이지에 산재하여 **원스톱 신청 불가**
- 영수증 스캔, 환급 추적 등 **실행 지원 기능 부재**
- 청년·MZ세대 사용자 접점에서 **인지도 낮음**

#### 온누리상품권 앱 / 지역사랑상품권 앱

각 혜택 프로그램별로 독립 앱이 존재. 온누리상품권 앱, 제로페이, 지역별 상품권 앱이 **파편화**되어 있어 사용자가 어떤 앱을 써야 할지 파악하기 어렵다.

## 3. 시장 공백 (Gap Analysis)

### 3.1 구조적 분리 문제

```
[여행 계획]     [보조금 신청]      [현지 소비]       [사후 정산]
야놀자/여기어때  구석구석/지자체웹   온누리앱/제로페이   영수증 직접 제출
     ↑                ↑                 ↑                  ↑
   앱 따로          앱 따로           앱 따로            앱 따로
```

사용자는 하나의 여행을 위해 **최소 4개 앱·웹사이트**를 오가야 한다. 이 단절이 혜택 인지 → 신청 → 소비 → 환급의 전환 경로 전체에 마찰을 만든다.

### 3.2 정보 비대칭

대부분의 여행자는 **떠난 뒤**에야 혜택을 인지한다. 숙박세일페스타 쿠폰이 선착순 소진되거나, 신청 기간이 지난 후 알게 되는 사례가 다수다. 혜택 캘린더·알림 기능을 갖춘 여행 앱이 없기 때문이다.

### 3.3 비수도권 여행 유인 부재

정부 혜택의 상당수는 **인구감소지역·비수도권** 방문을 전제조건으로 한다. 그러나 야놀자·여기어때는 수도권 및 인기 관광지 위주 노출 알고리즘을 사용, 비수도권 여행 동기 형성에 기여하지 못한다.

### 3.4 여행 문화의 변화

MZ세대를 중심으로 '효율적 소비'에 대한 관심이 높아지면서, 단순 예약 앱을 넘어 **총비용을 줄여주는 플랫폼**에 대한 수요가 증가하고 있다. 가성비 여행, 리워드 최적화 여행이 트렌드화되고 있음에도 이를 전문적으로 지원하는 앱은 부재하다.

## 4. TripMate의 포지셔닝과 차별성

### 4.1 서비스 포지셔닝

TripMate는 기존 여행 앱들이 점령한 **'예약' 버티컬**이 아닌, 아무도 채우지 않은 **'혜택 여정 전체'** 버티컬을 공략한다.

```
기존 앱들의 전장:           TripMate의 전장:
[숙소 예약 경쟁]           [여행 전~후 혜택 전체 경험]

야놀자 ──────────────┐      여행지 추천 (취향 분석)
여기어때 ─────────────┤  →     ↓
에어비앤비 ───────────┘   혜택 탐색 & 신청 가이드
                              ↓
                         현지 소비 유도 (온누리 등)
                              ↓
                         영수증 스캔 & 자동 환급 신청
                              ↓
                         사후 정산 리포트
```

### 4.2 핵심 차별화 포인트 5가지

#### ① 여행 취향 × 혜택의 결합
야놀자/여기어때는 "어느 숙소를 예약할까"에 집중하지만, TripMate는 "내가 받을 수 있는 혜택 안에서 어디를 가면 좋을까"로 **의사결정 출발점**을 바꾼다. 취향(자연관광, 맛집 등)과 혜택 조건(인구감소지역, 참여 숙소 등)을 함께 고려한 코스 추천은 현재 시장에서 유일하다.

#### ② 여행 전·중·후 연속성
단순 예약 완료로 끝나는 기존 앱과 달리, TripMate는 **계획 → 신청 → 소비 → 정산**의 전 여정을 하나의 앱에서 처리한다. 특히 OCR 영수증 스캔을 통한 환급 신청 자동화는 사용자 마찰을 대폭 줄인다.

#### ③ 비수도권·인구감소지역 특화
정부 혜택의 대부분이 비수도권·인구감소지역을 대상으로 하기 때문에, TripMate의 추천 알고리즘은 기본적으로 **혜택이 최대화되는 비수도권 목적지**를 우선 노출한다. 이는 상업적 알고리즘(인기 지역 우선)을 사용하는 기존 앱과 구조적으로 다르다.

#### ④ 예산 소진 실시간 알림
정부 보조금은 예산 소진형이 많아 조기 마감이 빈번하다. TripMate는 잔여 예산 비율을 실시간 표시하고 소진 임박 시 **선제적 알림**을 발송한다. 이는 기존 앱에서 전혀 제공되지 않는 기능이다.

#### ⑤ OTA 없는 중립적 정보 제공
야놀자·여기어때는 광고비를 지급한 숙소를 우선 노출하는 **유료 알고리즘** 구조를 가진다. TripMate는 공공데이터를 기반으로 하여 **광고 편향 없는 중립적 정보**를 제공한다.

## 5. 핵심 강점 분석

### 5.1 기술적 강점

| 강점 | 설명 |
|------|------|
| **OCR 영수증 스캔** | 전통시장·지역 상점 영수증을 카메라로 촬영하면 환급 대상 여부와 예상 환급액을 즉시 계산. 수작업 제출 프로세스를 앱 내에서 완결 |
| **취향 기반 AI 추천** | 여행 목적(자연관광, 맛집 등)과 기간을 입력하면 혜택 조건을 반영한 최적 코스를 자동 생성 |
| **공공데이터 통합** | TourAPI 260,000+ 관광 데이터를 기반으로 한 신뢰할 수 있는 정보 기반 |
| **실시간 혼잡도 표시** | 관광지별 현재 혼잡 수준 표시로 비수기·비수도권 여행 유인 강화 |

### 5.2 비즈니스적 강점

| 강점 | 설명 |
|------|------|
| **레드오션 회피** | 숙박 예약 플랫폼 경쟁에서 탈피, 아무도 없는 '혜택 여정 관리' 포지션 선점 |
| **정부 정책 수혜** | 내수 관광 활성화, 지역균형발전, 전통시장 살리기 등 정부 방향성과 일치 → 공공 협력 기회 |
| **높은 재방문 동기** | 환급 이력, 취향 프로필, 적립 리워드 등이 쌓이면서 앱 이탈 유인 감소 |
| **낮은 초기 인프라 비용** | 공공 API를 활용하므로 관광 데이터 구축 비용 없이 서비스 시작 가능 |

### 5.3 사회적 강점

- **지역 경제 활성화 기여**: 비수도권 여행 유도 → 지역 상권 소비 증가
- **정보 접근성 평등화**: 정부 혜택을 IT 친숙도가 낮은 사용자도 쉽게 활용
- **전통시장 디지털화**: OCR + 온누리상품권 연동을 통해 전통시장을 앱 경제에 편입

## 6. 약점 및 한계

### 6.1 서비스 완성도 측면

| 약점 | 설명 |
|------|------|
| **실제 예약 미지원** | 현재는 혜택 안내·코스 추천까지만 가능. OTA 연동 또는 자체 예약 기능 없이는 최종 전환 불가 |
| **실시간 데이터 부재** | 공공 API의 데이터는 갱신 주기가 일정하지 않아 폐업 업소나 행사 취소 정보가 늦게 반영됨 |
| **OCR 정확도 의존성** | 영수증 인식률은 인쇄 품질·조명에 따라 편차가 크며, 수기 영수증은 인식 자체가 어려움 |
| **정부 API 불안정성** | 지자체별 혜택 프로그램은 별도 API가 없는 경우가 많아, 데이터를 직접 크롤링하거나 수동 업데이트에 의존해야 함 |

### 6.2 사용자 경험 측면

| 약점 | 설명 |
|------|------|
| **혜택 조건의 복잡성** | "인구감소지역이 어딘지"를 사용자가 이미 알고 있다는 전제가 있으면 UX가 복잡해짐 |
| **취향 프로필 콜드 스타트** | 처음 사용자는 추천 결과의 정확도가 낮아 첫인상 저하 가능성 |
| **혜택 만료 후 공백** | 특정 기간에만 운영되는 혜택이 종료되면 앱 사용 동기가 급감할 수 있음 |
| **40대 이상 사용자 접근성** | 취향 선택 → 코스 추천 → 혜택 신청으로 이어지는 멀티스텝 플로우가 비교적 복잡함 |

### 6.3 비즈니스 측면

| 약점 | 설명 |
|------|------|
| **수익 모델 불명확** | 공공데이터 기반·비상업적 포지셔닝을 유지하면서 지속 가능한 수익 창출 방법이 명확하지 않음 |
| **대형 플랫폼 추격 리스크** | 야놀자나 네이버가 유사 기능을 도입할 경우 즉각적인 위협이 될 수 있음 |
| **혜택 정책 변경 리스크** | 정부 예산·정책에 따라 핵심 혜택 프로그램이 사라지거나 조건이 바뀔 경우 앱 가치 하락 |

## 7. 개선 방향 및 발전 가능성

### 7.1 단기 개선 (MVP → v1.0)

**① OTA 딥링크 연동**
야놀자·여기어때 API 또는 딥링크를 통해 TripMate에서 코스를 구성한 뒤 해당 숙소로 바로 이동·예약이 완결되도록 한다. 수익 모델로서 제휴 커미션을 검토할 수 있다.

**② 혜택 신청 프로세스 인앱 완결**
지자체 웹사이트로 이동하는 현재 방식에서, 주요 혜택 신청 폼을 앱 내 WebView로 임베드하거나 간편 신청 API를 연동하여 이탈률을 줄인다.

**③ 푸시 알림 시스템**
- 북마크한 혜택의 D-day 알림
- 선착순 오픈 시각 알림 (예: 숙박세일페스타 4/8 10:00)
- 잔여 예산 임계치 도달 알림

### 7.2 중기 개선 (v1.0 → v2.0)

**④ 지도 기반 혜택 가시화**
온누리상품권 가맹 전통시장, 지역사랑 휴가지원 가능 지역을 지도 레이어로 표현. 여행 동선 위에 '환급 가능 장소'를 오버레이하여 현지 소비 행동을 유도한다.

**⑤ 소셜 혜택 리뷰**
실제 혜택을 받은 사용자의 후기 (환급 금액, 소요 기간, 주의사항)를 앱 내 커뮤니티로 축적. 혜택 정보의 신뢰도와 SEO 자산을 동시에 확보한다.

**⑥ AI 개인화 고도화**
취향 프로필이 누적될수록 더 정교한 추천이 가능하도록 협업 필터링(Collaborative Filtering) 알고리즘을 도입. "비슷한 취향의 여행자들이 이 혜택을 활용한 코스" 형식의 추천을 제공한다.

### 7.3 장기 발전 방향 (v2.0+)

**⑦ B2G (Business to Government) 협력**
관광공사·지자체를 대상으로 혜택 집행 효율 분석 리포트를 제공하는 데이터 서비스로 확장. 정부 입장에서도 예산 대비 여행자 유입 효과를 실시간으로 측정할 수 있는 플랫폼이 필요하다.

**⑧ 지역 파트너십 플랫폼**
인구감소지역 지자체와 직접 파트너십을 맺어 독점 혜택 정보를 앱 내에서 가장 먼저 공개하는 구조를 만든다. 이는 다른 플랫폼이 모방하기 어려운 **데이터 해자(Data Moat)**를 형성한다.

## 8. 공공데이터 API 활용의 이점과 기대효과

### 8.1 한국관광공사 TourAPI 개요

TourAPI는 한국관광공사(KTO)가 공공데이터포털(data.go.kr)을 통해 제공하는 국내 최대 관광 정보 오픈 API다.

| 항목 | 내용 |
|------|------|
| 데이터 건수 | 약 260,000건 이상 |
| 제공 분류 | 관광지, 숙박, 음식점, 축제/공연, 레포츠, 문화시설, 쇼핑, 교통 등 15종 |
| 갱신 주기 | 실시간~주간 (분류에 따라 상이) |
| 저작권 | CC BY (출처 표시 후 자유 이용) |
| 제공 형식 | REST API (JSON/XML) |
| 지역 코드 | 전국 시·군·구 단위 areaCode 체계 |

### 8.2 TourAPI 활용 이점

#### ① 데이터 구축 비용 Zero
260,000건의 관광 데이터를 자체 구축하려면 수억 원의 비용과 수년의 시간이 필요하다. TourAPI를 활용하면 이 비용 없이 서비스 즉시 시작이 가능하다.

#### ② 정확성과 신뢰성 보장
한국관광공사가 직접 수집·검증한 데이터로, 민간 크롤링 데이터에 비해 폐업·휴업 정보 반영 속도가 빠르고 오류율이 낮다.

#### ③ 표준화된 지역 코드 체계
`areaCode` + `sigunguCode` 체계는 정부 혜택 데이터(지역사랑 휴가지원 대상 지역, 인구감소지역 목록 등)와 **동일한 행정 코드 체계**를 사용하므로, 관광 정보와 혜택 조건을 코드 기반으로 매핑하는 것이 용이하다.

#### ④ 혼잡도·실시간 데이터 확장
한국관광공사는 통신사 데이터 기반 관광지 혼잡도 API를 별도 제공 중이며, 이를 통해 "지금 한산한 여행지" 기능의 데이터 정확도를 크게 높일 수 있다.

#### ⑤ 법적 안전성
저작권 CC BY 라이선스로 상업적 이용이 가능하며, 데이터 무단 사용으로 인한 법적 리스크가 없다.

### 8.3 추가 활용 가능한 공공 API

| API | 제공 기관 | 활용 가능 데이터 |
|-----|----------|----------------|
| **전통시장 정보 API** | 중소벤처기업부 | 온누리상품권 가맹 전통시장 위치·영업 정보 |
| **인구감소지역 현황** | 행정안전부 | 지역사랑 휴가지원 신청 가능 지역 판별 |
| **문화재 정보 API** | 문화재청 | 역사·문화 여행 코스 데이터 보강 |
| **날씨 API** | 기상청 | 여행지 날씨 예보 표시 (현재 더미 데이터) |
| **대기오염 측정 API** | 환경부 | 야외 활동 최적 날짜 추천 |
| **대중교통 API** | 국토교통부 | 여행 코스 내 교통편 정보 제공 |

### 8.4 공공데이터 활용의 사회적 기대효과

#### 국민 편익 측면
- 정부 혜택 **인지→신청→환급 전환율** 상승 → 실질적 여행비 절감
- 디지털 취약계층도 간편하게 혜택 접근 가능한 **정보 접근성 평등화**

#### 지역 경제 측면
- 취향 기반 비수도권 추천으로 **수도권 쏠림 현상 완화**
- 인구감소지역 방문객 증가 → 지역 숙박·식음료·전통시장 매출 증대
- 온누리상품권 소비 확대를 통한 **전통시장 생태계 유지**

#### 정책 효과성 측면
- 정부 여행 보조금의 **사용률·집행 효율 제고**
- 사용자 행동 데이터를 통한 **다음 연도 혜택 설계 인사이트** 제공 가능
- 민관 협력 모델 사례로서 **공공데이터 개방 정책 정당성 강화**

### 8.5 TourAPI 데이터 구조와 TripMate 설계 연계

```
TourAPI 응답 필드         TripMate 활용
─────────────────────    ──────────────────────
contentId           →    관광지 고유 식별자
areaCode            →    지역사랑 휴가지원 자격 판별
sigunguCode         →    세부 지역 필터링
contenttypeid       →    카테고리 분류 (자연/문화/숙박 등)
cat1/cat2/cat3      →    여행 취향 매핑 (nature/history/food)
mlevel              →    지도 확대 수준
readcount           →    인기도 가중치 (혼잡도 역산 참고)
eventstartdate      →    축제·행사 일정 표시
bookingurl          →    OTA 예약 링크 연동
```

## 9. 결론

### 9.1 시장 기회 요약

국내 여행 앱 시장은 **숙박 예약 플랫폼의 경쟁 레드오션**과 **정부 혜택 정보의 파편화된 공급 구조** 사이에 거대한 공백이 존재한다. 이 공백은 단순히 앱이 없는 것이 아니라, 기존 앱들의 **비즈니스 모델이 구조적으로 채울 수 없는 영역**이다.

상업 플랫폼(야놀자, 여기어때)은 예약 커머스로 수익을 내야 하므로 광고비를 낸 인기 숙소를 우선 노출하는 알고리즘에서 벗어날 수 없다. 반면 TripMate는 **공공데이터 기반의 중립적 혜택 여정 플랫폼**으로서 이 공백을 메울 수 있는 구조적 강점을 갖는다.

### 9.2 TripMate의 성공 조건

1. **혜택 데이터의 실시간성 확보**: 공공 API 갱신 주기 한계를 자체 모니터링으로 보완
2. **예약 전환 연결고리 마련**: OTA 딥링크 또는 자체 예약 기능으로 앱 내 완결성 확보
3. **사용자 리텐션 설계**: 환급 이력 누적, 취향 프로필 발전으로 앱 이탈 방지
4. **정부·지자체와의 파트너십**: 독점 혜택 정보 채널로서의 공식 지위 획득

### 9.3 최종 평가

> TripMate는 **"여행 예약 앱"이 아닌 "국내 여행 혜택의 인프라"** 를 지향하는 서비스다.
> 기존 플랫폼이 경쟁하는 시장 위가 아닌, 그 아래 레이어에서 작동하는 **공공재적 서비스 모델**은 상업 플랫폼이 모방하기 어렵고, 정부 협력을 통해 진입장벽을 높일 수 있다.
> 공공데이터 API를 핵심 인프라로 삼은 선택은 비용 구조와 신뢰성 양면에서 유효하며, 이를 바탕으로 **여행지 추천 → 혜택 신청 → 현지 소비 → 사후 정산**의 완결된 사용자 여정을 구현하는 것이 TripMate의 핵심 과제이자 차별적 가치다.
